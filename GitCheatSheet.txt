WORKING DIRECTORY (LOCAL):
1. Create files or directories as per need in the directory you are working in
OR 
create repository in GitHub with the required files and then clone it on your desired location in File Explorer
"git clone <GitHub URL>" -> (Will download the clone of the repository)
2. "git init" -> (TO initialize git to then keep track of all the changes and modifications in the directory)

STAGING:
1. Add/delete/update files or directories and make modifications in them as per the requirements
2. Once finalized and you are ready to move to the next step
3. "git status" -> (TO check all the changes done whether its modified, added or deleted)
4. Once checked or reviewed then you can add it in temporary repository which allows you to revert your decision
before finally committing to the changes
"git add <Filename>" -> (To stage the specific file you modified in the current directory)
"git add --all" or "git add -A" -> (To stage all the files that are changed in the whole project/directory)
"git add ." -> (To stage all the files and sub-directories within the current directory)
"git add *" -> (To stage all the files and sub-directories except for the deleted ones)
"git add *.txt" -> (To stage all the ".txt" files within the current directory)

2. "git reset" -> (When you want to undo your changes, so you remove it from the staging part)

3. "DEL <filename>" then "git add ." -> (To delete a file and then stage it to track it)
OR "git rm <filename>" -> (Delete and stage it in a single command)
"git rm -r <FOLDER>" -> (To delete all the files and subdirectories within that folder recursively)
"git rm -f <filename>" -> (To forcefully delete a file if git doesn't let you due to uncommitted changes detected)

"git reset --hard" -> (This will recover the deleted files but may remove the untracked/uncommitted files)
"git reset --cached <filename>" -> (This removes the file from the staging area but keeps in the working directory)

LOCAL REPOSITORY:
1. Once staged all the changes
"git commit -m "<MESSAGE>"" -> (To commit the changes made)
Note: This command does not save it on the remote repository. It just sits between your working directory 
and remote repository (local repository)

2. "git reset HEAD~" -> (To move back from the local repository stage to working directory (unstage))

BRANCHING:
1. Main branch is the default MASTER branch in GitHub
2. Multiple branches can be made to work independently or experiment and then merge everything back to the main branch
to reduce mistakes in main branches and increase efficiency
"git branch <BRANCH-NAME>" -> (To create a new branch)

"git checkout <BRANCH-NAME>" -> (To move from the main or current branch to other branch)

MERGING:
3. Then you can make your own changes in the existing files or new files and then merge it in the main 
branch
Note: Make sure you commit first before merging!
"git merge <BRANCH-NAME> -m <MESSAGE> -> (To merge the changes in a branch from another branch)

MERGE CONFLICT:
BUT THERE CAN BE MERGE CONFLICTS (when two branches have changes in the same file and same path)
Git gets confused on which change to keep and which to discard. So we manually solve it and then merge it

GO TO A PREVIOUS VERSION:
"git log --oneline" -> (displays all the commits made with their ids)
"git checkout <COMMIT-ID>" -> (Allows you to go back to the previous version only to view)

COMPARING TWO FILES (LINES ADDED OR DELETED):
"git diff <NEW-COMMIT-ID> <OLD-COMMIT-ID>" -> (GREEN/BLUE SHOWS ADDED AND RED SHOWS DELETED)

PUSH:
"git push origin main" -> (where origin = the local repository AND main = branch you want to push in)

FETCH:
"git fetch" -> (You made the changes in the GitHub repository and you want to reflect it in your working directory)
NOTE: fetch wont make the changes in your working directory unless you merge it
"git merge"

PULL:
"git pull" -> (It fetches and merges both in the working directory)

RESTORE (A LIFE SAVER):
1. Lets say you change, modify or try a new approach and change a lot of files
Midway or in the end you decide this is not working so you want to undo all the new changes
MANUALLY CHANGING IT IS ALMOST IMPOSSIBLE!
"git restore <filename> -> (Restores the file to the previous commit made)
"git restore <FOLDER> -> (Restores all the files in the directory/folder to its previous commit)
"git restore ." -> (Restore every single file in the entire project)
"git restore --staged <filename>" -> (Removes the file/s from the staging area back to working directory)

STASH:
1. Git does not allow you to switch to another branch before committing the current changes.
2. Stash is done when you want to set aside your current unfinished work without committing and work
on another branch

"git stash" -> (Allows you to switch)
Note: It will hide the changes you made in the current directory but it is not lost
just moved to a temporary area

"git stash pop" -> (To get back the unfinished work you stashed and removes/drops the stash from the stack)
"git stash apply" -> (to get back the changes made before stashing but it still keeps the stash in the stack)
"git stash list" -> (lists all the stash number and ID in the stack (newest on top))

can also access it like a stack -> "git stash pop stash@{0}"

"git stash drop" -> (Drops or deletes the stash in the stack)

REVERT: (Works similar to restore or reset but creates a new commit so that its traceable)
1. it undo the changes made in the file to the previous commit made, but instead of deleting that
old commit, it creates a new one that reverse those changes 
"git log --oneline" -> (To get the commit ID)
"git revert <COMMIT-ID>" -> (To revert or go back to that commit but still preserving the current commit)

REBASE: (Similar to merge)
1. Lets say you make a new branch and start creating and modifying in that branch, and someone else
makes changes in the main branch or some other branch that you need to include in your current working branch
Now using "git merge <branch-name>" is one option, but it creates an extra log of merge which makes log history messy

So, we use REBASE. It does not create an extra log and keeps log history clean and linear.
"git rebase <branch-name>"
NOTE: Even though rebase is more powerful and easier, it must not be done in public repositories
or you should do it after informing, as it rewrites existing commit history, and can even change the commit IDs
making it hard to sync or pull

PULL REQUEST (PR):
1. Used in GitHub to request to merge your current branch in another branch (main)
Its like telling someone that I have implemented the feature or changes, review it, and if everything is fine
then merge them into the main branch
